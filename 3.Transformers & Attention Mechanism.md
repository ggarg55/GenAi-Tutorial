# Transformers & Attention Mechanism: A Complete Tutorial

This tutorial explains the Transformer architecture, focusing on the attention mechanism that powers modern AI models like GPT, BERT, and many others. We'll use simple analogies and step-by-step examples to make these concepts unforgettable.

---

## 🎯 Why Transformers?

Before transformers, models like RNNs processed sequences one token at a time:

```
RNN: "The" → "mouse" → "cannot" → "kill" → ...
```

This was **slow** because it couldn't be parallelized.

Transformers introduced a revolutionary idea: **process all tokens in parallel**. They achieve this through the **attention mechanism**, which allows the model to focus on the most relevant parts of the input regardless of their position.

---

## 🔍 What Is Attention?

Let's start with an example:

> **"The mouse cannot kill the lion because it was too"**

What should come next? You probably thought "small" or "afraid". How did you know? Your brain automatically focused on certain words:

- **mouse** (who is the subject)
- **kill** (what action)
- **lion** (who is the object)

**Attention is exactly this focusing mechanism.** It lets the model weigh the importance of each word when making a prediction.

---

## 🧮 The Attention Formula

The core of the Transformer is the **scaled dot‑product attention**:

```
Attention(Q,K,V) = softmax( QKᵀ / √d_k ) V
```

Let’s break it down with a simple example.

### Step 1: Represent Words as Vectors (Embeddings)

Each word is converted into a vector of numbers. The vectors are of the same length so we can compare them.

```
"the"    → [0.2, 0.5, 0.1, 0.8]
"mouse"  → [0.9, 0.3, 0.7, 0.2]
"cannot" → [0.4, 0.6, 0.3, 0.5]
"kill"   → [0.8, 0.2, 0.9, 0.1]
"lion"   → [0.7, 0.8, 0.2, 0.4]
"it"     → [0.3, 0.4, 0.5, 0.6]
"was"    → [0.5, 0.5, 0.5, 0.5]
"too"    → [0.6, 0.1, 0.4, 0.9]   ← this is our query
```

### Step 2: Create Query, Key, and Value

We multiply the embedding vector by three learned matrices to obtain **Query (Q)**, **Key (K)**, and **Value (V)** for each token:

```
Q = X · W_Q
K = X · W_K
V = X · W_V
```

- **Query (Q)**: what we are looking for (e.g., the word "too")
- **Key (K)**: a label for each token (what it offers)
- **Value (V)**: the actual information carried by the token

> **Analogy:** Think of a library. The query is your search term. The key is the book’s title on the shelf. The value is the book’s content.

### Step 3: Compute Similarity (QKᵀ)

We calculate how similar the query is to every key using the dot product:

```
Similarity(query, key) = query · key
```

For our sentence, the similarity between the query "too" and each key might look like:

| Word   | Similarity Score |
|--------|------------------|
| mouse  | 0.9 (high)       |
| kill   | 0.8 (high)       |
| lion   | 0.6 (medium)     |
| cannot | 0.7 (high)       |
| it     | 0.4 (low)        |
| was    | 0.2 (very low)   |
| the    | 0.1 (very low)   |

These scores tell us which words are most relevant to predicting the next word after "too".

### Step 4: Scale by √d_k

We divide by the square root of the key vector dimension `d_k`. Why? If the vectors are large, the dot products can become huge, making the softmax too “peaked” and hindering learning. Scaling keeps values in a stable range.

> **Intuition:** If you divide by `d_k`, the numbers shrink too much and the softmax becomes flat. √d_k gives a balanced scale.

### Step 5: Apply Softmax

Softmax turns the scores into probabilities that sum to 1:

```
[0.1, 0.9, 0.7, 0.8, 0.6, 0.4, 0.2, 0.5]
   ↓
[0.02, 0.28, 0.18, 0.22, 0.12, 0.08, 0.03, 0.07]
```

These are **attention weights**. They tell the model how much to focus on each word.

### Step 6: Multiply with Values (V)

We multiply each word’s value vector by its attention weight:

```
weighted_value(word) = attention_weight(word) × value(word)
```

### Step 7: Sum Everything

We add all weighted values to get a single vector that represents the **attended information**. This vector is then used to predict the next word.

## 🔄 Multi‑Head Attention

A single attention head can capture one type of relationship. But language is complex: a word may be related to others in multiple ways (grammar, semantics, proximity, etc.).

**Multi‑head attention** runs several attention mechanisms in parallel, each with its own learned projection matrices. Their outputs are concatenated and projected to form the final result.

> **Analogy:** Imagine having multiple experts analyzing the same sentence. One focuses on nearby words, another on subject‑verb agreement, another on long‑range dependencies. Together they form a richer understanding.

Mathematically:

```
MultiHead(Q,K,V) = Concat(head₁, …, headₕ) W⁰
where headᵢ = Attention(QWᵢ^Q, KWᵢ^K, VWᵢ^V)
```

---

## 🚫 Casual Masking (for Decoders)

When training a Transformer to generate text, we want it to predict the next word **without seeing future words**. Casual masking ensures that at each step, the model can only attend to previous tokens.

For example, given the sentence "The mouse cannot kill the lion", we train the model to predict:

- After "The" → "mouse"
- After "The mouse" → "cannot"
- After "The mouse cannot" → "kill"
- … and so on.

All these predictions are made in parallel by **masking** the future tokens during training.

## 🏗️ The Transformer Block

A Transformer is built from blocks that contain:

1. **Multi‑Head Attention** – attends to the input.
2. **Add & Layer Norm** – residual connection + layer normalisation.
3. **Feed‑Forward Network** – a simple MLP applied to each position separately.
4. **Add & Layer Norm** – another residual connection.

```
          Output
             ↑
    ┌────────┴────────┐
    │ Layer Normalisation │
    └────────┬────────┘
             ↑
    ┌────────┴────────┐
    │ Feed‑Forward Network │
    └────────┬────────┘
             ↑
    ┌────────┴────────┐
    │ Layer Normalisation │
    └────────┬────────┘
             ↑
    ┌────────┴────────┐
    │ Multi‑Head Attention │
    └────────┬────────┘
             ↑
          Input (Q,K,V)
```

Multiple such blocks can be stacked to create deep models.

## 📍 Positional Encoding

Since the Transformer processes tokens in parallel, it has no built‑in notion of word order. We need to inject position information.

**Solution:** Add a **positional encoding** to each token’s embedding. The original paper uses sine and cosine functions of different frequencies:

```
PE(pos, 2i)   = sin(pos / 10000^(2i/d_model))
PE(pos, 2i+1) = cos(pos / 10000^(2i/d_model))
```

This creates a unique pattern for each position and allows the model to learn relationships based on distance.

> **Simple analogy:** Each word gets a nametag saying “I’m word #3 in the sentence.” The encoding is added to the word embedding, so the model knows both what the word is and where it appears.

---

## 🌟 Key Takeaways

1. **Attention = Focus.** It allows the model to weigh the importance of each token when making a decision.
2. **Query, Key, Value** are learned projections that enable the model to compare tokens and retrieve relevant information.
3. **Multi‑Head Attention** captures different types of relationships by running multiple attention mechanisms in parallel.
4. **Positional Encoding** preserves order information.
5. **Transformers are parallelizable**, which makes them efficient and scalable to huge datasets.
6. **Modern AI (GPT, BERT, etc.)** is built on this foundation.
