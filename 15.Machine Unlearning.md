## 📖 Introduction

Imagine you’ve trained a powerful AI model – a recommendation engine, a facial recognition system, or a resume screener. It’s accurate, fast, and ready for production. Then a user asks: “Delete my data. I want to be forgotten.”

You delete their file from the database, but the model still contains their influence. Its weights were shaped by that data during training. Simply removing a file doesn’t make the model forget.

Welcome to **Machine Unlearning** – the emerging field that teaches AI how to forget specific data points efficiently and verifiably.

In this tutorial, you’ll learn:

- What machine unlearning is and why it matters
- Real-world scenarios where forgetting is essential
- The technical challenges behind unlearning
- Formal definitions and evaluation techniques
- Ethical considerations for real‑world deployment

---

## 🧠 What is Machine Unlearning?

Traditional machine learning is all about **learning** – extracting patterns from data and encoding them into model weights. Unlearning is the reverse: **removing the influence of a specific subset of training data** without retraining the entire model from scratch.

Think of it like this:

- **Learning** = baking a cake with all ingredients.
- **Unlearning** = removing a single egg from the finished cake. You can’t just pick it out – you need to fundamentally alter the recipe.

The goal of unlearning is to produce a new model that behaves *as if* the forgotten data was never used during training, but without the astronomical cost of retraining.

---

## ❓ Why Do We Need Models to Forget?

The transcript highlights four major reasons. Let’s explore each with a concrete example.

| Reason | Example |
|--------|---------|
| **Security** | An adversarial example (e.g., a specially crafted image) tricks a face‑recognition system. Once detected, we want the model to unlearn that vulnerability. |
| **Privacy** | Under GDPR, users have the “right to be forgotten.” If a user deletes their account, the AI trained on their data must also forget them. |
| **Usability** | A recommendation system keeps suggesting phones after you’ve already bought one. Unlearning your “phone‑shopping” behavior makes the experience relevant again. |
| **Fairness** | A hiring model learns gender or racial bias from historical data. Unlearning biased samples helps create a fairer system. |

## 🔥 The Core Challenges

Unlearning is hard. Here’s why:

### 1. Stochastic Training Process
Modern models are trained with stochastic algorithms like **SGD**. Data is shuffled, updates are random. We don’t know how much a single data point contributed to the final weights. Removing one sample is like removing a single drop from the ocean – the impact is invisible.

### 2. Interdependence of Data Points
Data points don’t act in isolation. Removing one can affect how the model learns from the remaining data. The ripple effect is unpredictable.

### 3. Catastrophic Unlearning
If unlearning is done poorly, the model may lose too much knowledge and become useless. The unlearned model could be significantly worse than simply retraining from scratch – defeating the purpose.

## 🧩 Formalizing the Problem

To discuss unlearning precisely, we need a common vocabulary. Let’s define:

- **Z** – The space of all possible examples (e.g., every houseplant image).
- **Z*** – The set of all possible training datasets.
- **D** ∈ **Z*** – Your actual training dataset.
- **H** – The hypothesis space (all possible models).
- **A** – The learning algorithm (e.g., SGD with a ResNet).  
  `A(D)` = a model trained on **D**.
- **D_f** ⊆ **D** – The **forget set** (data to unlearn).

An **unlearning algorithm** `U` takes the original model `A(D)`, the original dataset **D**, and the forget set **D_f**, and produces a new model:

```
U(A(D), D, D_f) ∈ H
```

The goal is for this unlearned model to be nearly indistinguishable from a model retrained from scratch on `D \ D_f` (the dataset with the forget set removed).

---

## 🎯 Exact Unlearning – The Gold Standard

A unlearning algorithm is **exact** if:

```
Distribution of U(A(D), D, D_f) = Distribution of A(D \ D_f)
```

In plain English: the unlearned model should be statistically identical to a model trained from scratch on the remaining data. No external test can tell them apart.

**Why this matters:** It provides a strong privacy guarantee. If you claim a user’s data is gone, an exact unlearning algorithm ensures the model reflects that truth.

**The catch:** Exact unlearning is computationally expensive, often requiring techniques like *partitioning* or *differential privacy*. In practice, many systems settle for **approximate unlearning**, where the difference is small enough to be acceptable.

Users can ask to forget data at different granularities:

| Type | Description | Example |
|------|-------------|---------|
| **Sample Deletion** | Forget a single data point. | A user wants one of their pictures removed. |
| **Feature Deletion** | Forget all information related to a feature. | “Forget the gender column from my hiring dataset.” |
| **Class Deletion** | Forget an entire category. | In face recognition, a person opts out – delete all images of that identity. |

## ✅ Design Requirements for Good Unlearning

A production‑ready unlearning system should satisfy three key properties:

1. **Verifiability** – We must be able to *check* that the data was truly forgotten. (We’ll see how in the next section.)
2. **Consistency** – The unlearned model should perform as well as a retrained model on the remaining data.
3. **Accuracy** – Accuracy may drop (less data), but it should be *comparable* to retraining from scratch.

---

## 🔍 How to Validate Unlearning

If a company says “We unlearned your data,” how can you verify it? Researchers have developed attack‑based tests to probe the model’s memory.

### 1. Membership Inference Attack

**Goal:** Determine whether a specific data point was used in training.

**How it works:**

1. **Train shadow models** – models of the same architecture on different datasets (similar to but disjoint from the original).
2. **Train an attack model** – this model learns to predict, from the output of a shadow model, whether a given data point was in its training set.
3. **Test the target model** – feed the forgotten data point into the attack model. If it predicts “member”, then unlearning failed.

If the attack cannot distinguish the forgotten point from unseen data, unlearning succeeded.

### 2. Backdoor Attack

**Goal:** Intentionally “poison” the training data to test if the poison is removed.

**How it works:**

1. **Inject a backdoor** – e.g., add a small invisible trigger to all images of class A and mislabel them as class B.
2. **Train the model** – it learns the backdoor.
3. **Request unlearning** – ask the model to forget the backdoored samples.
4. **Test** – present a new sample with the trigger. If the model now predicts class A (the true label), the backdoor is gone → successful unlearning. If it still predicts class B, the backdoor remains → failure.

## ⚖️ Ethical Considerations & Versioning

A crucial point to consider is: what about older versions of the model? If you keep a snapshot that was trained on the user’s data, you haven’t truly removed their influence.

**The ethical stance:** When a user requests deletion, you must delete **all versions** of the model that carry the effect of their data. Keeping old models is not acceptable – it’s a privacy violation and often illegal.

## 💡 Putting It All Together: A Simple Code Example

To make unlearning concrete, let’s implement a toy example in PyTorch. We’ll train a binary classifier on synthetic data, then unlearn one class using **gradient ascent** on the forget set – a common approximate unlearning technique.

```python
import torch
import torch.nn as nn
import torch.optim as optim
from torch.utils.data import DataLoader, TensorDataset

# 1. Create synthetic data (two Gaussian blobs)
torch.manual_seed(42)
X = torch.cat([torch.randn(500, 2) + torch.tensor([-2, -2]),
               torch.randn(500, 2) + torch.tensor([2, 2])])
y = torch.cat([torch.zeros(500), torch.ones(500)])  # class 0 and class 1

# Split into original training (90%) and forget set (10% of class 1)
train_X, train_y = X[:900], y[:900]
forget_X, forget_y = X[900:], y[900:]  # all belong to class 1

# 2. Train original model (simple logistic regression)
model = nn.Linear(2, 2)  # 2 classes
criterion = nn.CrossEntropyLoss()
optimizer = optim.SGD(model.parameters(), lr=0.01)

train_dataset = TensorDataset(train_X, train_y.long())
train_loader = DataLoader(train_dataset, batch_size=32, shuffle=True)

for epoch in range(50):
    for batch_X, batch_y in train_loader:
        optimizer.zero_grad()
        outputs = model(batch_X)
        loss = criterion(outputs, batch_y)
        loss.backward()
        optimizer.step()

print("Original model trained.")

# 3. Unlearn the forget set using gradient ascent
unlearn_loader = DataLoader(TensorDataset(forget_X, forget_y.long()), batch_size=32)

# Use a higher learning rate for unlearning
unlearn_optimizer = optim.SGD(model.parameters(), lr=0.1)

# Perform a few epochs of gradient ascent (maximize loss on forget set)
for epoch in range(10):
    for batch_X, batch_y in unlearn_loader:
        unlearn_optimizer.zero_grad()
        outputs = model(batch_X)
        loss = criterion(outputs, batch_y)
        # Negative gradient ascent: we want to increase loss, not decrease it
        (-loss).backward()
        unlearn_optimizer.step()

print("Unlearning performed.")

# 4. Validate: check accuracy on forget set before and after
with torch.no_grad():
    outputs = model(forget_X)
    _, pred = torch.max(outputs, 1)
    before_accuracy = (pred == forget_y).float().mean().item()
    print(f"Accuracy on forget set before unlearning: {before_accuracy:.4f}")

    # After unlearning, model should perform poorly on forget set
    outputs = model(forget_X)
    _, pred = torch.max(outputs, 1)
    after_accuracy = (pred == forget_y).float().mean().item()
    print(f"Accuracy on forget set after unlearning: {after_accuracy:.4f}")

# 5. Membership inference (simple): compare loss on forget set vs. held-out set
held_out = TensorDataset(X[950:], y[950:].long())
with torch.no_grad():
    loss_forget = criterion(model(forget_X), forget_y.long()).item()
    loss_held = criterion(model(held_out.tensors[0]), held_out.tensors[1]).item()
    print(f"Loss on forget set: {loss_forget:.4f}, Loss on held-out: {loss_held:.4f}")
```

**What this code does:**

- Trains a model on 900 points.
- Takes 100 points (all class 1) as the forget set.
- Unlearns them by maximizing the loss on those points (gradient ascent).
- After unlearning, the model performs poorly on the forget set – the influence is reduced.
- The membership inference check (loss comparison) shows that the forget set is now “harder” for the model, indicating it has been partially unlearned.

This is a simple approximation; real‑world unlearning requires more sophisticated techniques like **SISA** (Sharded, Isolated, Sliced, Aggregated) or **differential privacy**.
