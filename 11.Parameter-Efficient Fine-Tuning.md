## Introduction

Large language models (LLMs) have achieved remarkable performance across a wide range of tasks. However, adapting these massive models to specific downstream applications is challenging. Full fine-tuning—updating every parameter of the model—requires enormous computational resources and storage, making it impractical for most developers.  

This tutorial introduces two powerful techniques that enable efficient fine-tuning of LLMs: **LoRA** (Low-Rank Adaptation) and **QLoRA** (Quantized Low-Rank Adaptation). These methods drastically reduce the number of trainable parameters and memory footprint, allowing you to adapt models like GPT‑3 or Llama on consumer hardware.

---

## The Problem with Full Fine-Tuning

Suppose you have a pre-trained model with **175 billion parameters** (like GPT‑3). To fine-tune it for a specific task, you would need to:

- Update all 175 billion parameters.
- Store a separate copy of the model for each task.
- Use expensive hardware (multiple GPUs) and significant time.

If you have ten different tasks, you end up with ten full‑size models, each consuming hundreds of gigabytes of storage. This is neither efficient nor scalable.  

We need a method that:
- Keeps the original model fixed (frozen).
- Only trains a small number of additional parameters.
- Allows easy switching between tasks without storing multiple copies of the entire model.

LoRA and QLoRA were designed to meet these requirements.

## LoRA: Low-Rank Adaptation

### A Motivating Example

Consider a simple weight matrix of size **20 × 20**. It contains 400 parameters.  
Instead of updating all 400 values during fine‑tuning, we can represent the update as the product of two much smaller matrices:

- A column vector of size 20 × 1
- A row vector of size 1 × 20

Multiplying these two vectors gives a 20 × 20 matrix, but the total number of parameters is only **20 + 20 = 40**.  
This is a 90% reduction in trainable parameters!

This is the core idea behind LoRA: **any weight update can be approximated by a low‑rank decomposition**.

### Mathematical Formulation

Let **W** be a pre‑trained weight matrix of dimension **d × k**. In full fine‑tuning, we would update **W** to **W + ΔW**.  
LoRA proposes to constrain **ΔW** to be low‑rank:

\[
\Delta W = B \cdot A
\]

where:
- \(B\) has dimension \(d \times r\)
- \(A\) has dimension \(r \times k\)
- \(r\) is a small integer (the **rank**) with \(r \ll \min(d, k)\)

During fine‑tuning, **W** remains **frozen**—its values are never updated. Only the small matrices \(A\) and \(B\) are trained.

### How LoRA Integrates into a Model

The forward pass of a layer that originally computed \(h = Wx\) becomes:

\[
h = Wx + \Delta W x = Wx + (B \cdot A)x = Wx + B(Ax)
\]

This can be visualised as:

```
          Input x
             |
    +--------+--------+
    |                 |
    v                 v
   Wx               Ax
(frozen)             |
                     v
                    B
                     |
                     v
   +--------+--------+
   |        |
   v        v
   Wx + B(Ax)
   |
   v
   Output h
```

- The original path through **W** remains untouched.
- A parallel branch passes the input through \(A\) and then \(B\).
- The outputs of both branches are added.

Because the original weights are frozen, the model retains its pre‑trained knowledge while learning task‑specific adjustments through the small matrices \(A\) and \(B\).

### Initialization and Training

- **Matrix \(A\)** is initialized with random Gaussian values.
- **Matrix \(B\)** is initialized with zeros.

With this initialization, \(\Delta W = B A = 0\) at the start, so the model behaves exactly like the pre‑trained version. During training, the gradients update only \(A\) and \(B\).

### Important Properties of LoRA

1. **Applied to all weight matrices**  
   LoRA can be applied to every dense layer in the model (e.g., attention projections, feed‑forward layers). The same principle holds for each.

2. **The rank \(r\) controls expressiveness**  
   - Very small \(r\) → fewer trainable parameters, but possibly limited capacity.
   - Larger \(r\) → more capacity, but eventually converges to full fine‑tuning (when \(r\) equals the full rank of \(\Delta W\)).
   - Choose \(r\) based on the task complexity—often values like 4, 8, 16 are sufficient.

3. **No inference latency overhead**  
   Because \(Wx\) and \(B(Ax)\) can be computed in parallel, the additional computation is minimal. With efficient hardware, latency remains virtually unchanged.

4. **Effortless task switching**  
   Keep a single copy of the frozen base model. For each task, store only its specific \((A, B)\) matrices (which are tiny). To switch tasks, simply swap the LoRA parameters. To revert to the base model, ignore the LoRA branch.

### Why Does LoRA Work?

Research has shown that large language models possess a **low intrinsic dimension**. Even though the weight matrices are full‑rank, the directions in which the model can be effectively adapted lie in a much smaller subspace. LoRA exploits this by learning updates that are constrained to that subspace.

For GPT‑3 (175B parameters), LoRA can reduce the number of trainable parameters to as little as **0.1%** of the total—around **175 million** instead of 175 billion.  
- Gradients for LoRA occupy about **567 MB**.
- The LoRA parameters themselves require only **26 MB** of storage.

## QLoRA: Quantized LoRA

Even with LoRA, we still need to load the entire frozen base model into memory. A 65 billion parameter model in 16‑bit precision requires roughly **130 GB**—too much for a single GPU.  
**QLoRA** addresses this by quantizing the base model, dramatically reducing its memory footprint.

### What is Quantization?

Quantization converts numerical values from a high‑precision format to a lower‑precision one. Common formats:

- **32‑bit floating point (FP32)**: high precision, large memory (4 bytes per number).
- **16‑bit floating point (FP16/BF16)**: lower precision, half the memory.
- **8‑bit integer (INT8)**: even lower precision, a quarter of the memory.

A simple quantization formula for converting a tensor \(X\) (FP32) to INT8 is:

\[
X_{\text{int8}} = \text{round}\left(127 \cdot \frac{X}{\max(|X|)}\right)
\]

The scaling factor \(c = 127 / \max(|X|)\) ensures values fit in the range \([-127, 127]\).

To dequantize back to FP32:

\[
X_{\text{fp32}} \approx X_{\text{int8}} \cdot \frac{\max(|X|)}{127}
\]

### The Problem with Naive Quantization

If we apply the same scaling factor to an entire tensor, different FP32 values can collapse to the same INT8 value.  

For example, π ≈ 3.1415927. In 16‑bit (with 3 decimal places), both 3.1415927 and 3.1414927 become 3.141—they are indistinguishable. This information loss can degrade model performance.

### Blockwise Quantization

To preserve more information, **blockwise quantization** divides the tensor into smaller blocks and quantizes each block independently with its own scaling factor.

```
Input Tensor (size N)
        │
        ▼
   Divide into M blocks
   each of size B
        │
        ▼
   For each block:
     - Compute scaling factor c = 127 / max(block)
     - Quantize block: round(c * x)
```

Because each block has its own \(c\), values that would have merged in a global quantization can now be distinguished. This technique is essential for maintaining model quality when quantizing to 4‑bit or 8‑bit.

### Putting It Together: QLoRA

QLoRA combines three key ideas:

1. **Quantize the base model** to 4‑bit using blockwise quantization. This dramatically reduces memory usage.
2. **Freeze the quantized base model**.
3. **Add LoRA adapters** (\(A\) and \(B\)) in full 16‑bit precision on top of the quantized layers.

During training:
- The quantized base model stays frozen.
- Only the 16‑bit LoRA parameters are updated.

This approach allows fine‑tuning a **65 billion parameter model** on a **single 48 GB GPU** while preserving the full precision of the LoRA updates.

**Memory breakdown for a 7B model with QLoRA:**
- 4‑bit base model: ~3.5 GB
- LoRA parameters (16‑bit): ~26 MB
- Gradients: ~567 MB

## Summary and Key Takeaways

| Concept | Key Idea | Benefits |
|---------|----------|----------|
| **LoRA** | Represent weight updates as low‑rank matrices \(B \cdot A\); freeze original weights. | Reduces trainable parameters by 90%+, enables task switching without storing full models. |
| **QLoRA** | Quantize base model to 4‑bit using blockwise quantization; add LoRA in 16‑bit. | Cuts base model memory by ~75%, allowing fine‑tuning of 65B models on a single GPU. |

Both techniques rely on the observation that large models have a low intrinsic dimension for adaptation. By learning only the essential adjustments, we achieve efficient fine‑tuning without sacrificing performance.

**Practical implications:**
- You can fine‑tune models like Llama 2 7B on a consumer GPU.
- You can store dozens of fine‑tuned adapters for different tasks while keeping a single copy of the base model.
- Inference latency remains low because the extra computation is parallelizable.
